# 📝Kafka Introduction

## 1. 什麼是 Kafka？
* **一句話定義**：一個分布式的「事件串流平台」，就像是系統之間的**數位中央神經系統**。
* **核心價值**：負責在高併發環境下，安全、可靠地傳遞與儲存海量數據。

## 2. 核心四大角色 (Roles)
| 角色 | 術語 | 比喻 | 職責 |
| :--- | :--- | :--- | :--- |
| **生產者** | **Producer** | 記者/發文者 | 負責產生數據（訊息），並發送到 Kafka。 |
| **代理伺服器** | **Broker** | 報社節點/伺服器 | 負責接收、儲存訊息，並讓消費者讀取。 |
| **主題** | **Topic** | 報紙版面/分類 | 數據的分類標籤（如：訂單資訊、點擊記錄）。 |
| **消費者** | **Consumer** | 讀者/訂閱者 | 從指定的 Topic 抓取數據並進行後續處理。 |

## 3. Kafka 的靈魂：為什麼它不是普通的 Queue？
這是 Kafka 最特別的地方，也是它強大的原因：

* **Log 結構（日誌型儲存）**：
    * **傳統 Queue**：像「領飲料」，領完（讀完）資料就從隊伍中消失。
    * **Kafka Log**：像「寫日記」或「錄影帶」。數據寫入後是**永久或長久保存**的。
* **可重複消費 (Replayable)**：因為資料讀完不會消失，不同的系統可以在不同時間點「重讀」同份資料。
* **Retention Policy (保留政策)**：可以設定資料要留 7 天還是留 100GB，時間或容量到了才會刪除。

## 4. 關鍵技術特性
1.  **高吞吐量 (High Throughput)**：利用磁碟順序讀寫，每秒可處理百萬級訊息。
2.  **水平擴展 (Scalability)**：資料量大時，增加 Broker 數量即可分擔壓力。
3.  **分區與重平衡 (Partition & Rebalance)**：
    * **Partition**：把一個 Topic 切成好幾塊（像超市多開幾個收銀櫃檯）來加速處理。
    * **Rebalance**：當消費者變多或變少時，Kafka 會自動重新分配誰該負責哪個櫃檯。

## 5. 快速對比：Kafka vs. Redis
* **Redis**：
    * 像**「便利貼」**。
    * 存放在記憶體，極快但不安全（斷電易失）。
    * 適合：暫存、快取、簡單的傳話。
* **Kafka**：
    * 像**「圖書館/存證信函」**。
    * 存放在磁碟，極穩且具備持久性。
    * 適合：大數據處理、金融交易記錄、系統日誌收集。

## KRaft v.s. ZooKeeper

從 **ZooKeeper** 轉向 **KRaft** (Kafka Raft)，被譽為 Kafka 歷史上最重大的變革。
簡單來說，這是一場**「去中心化」**與**「簡化架構」**的革命。

---

### 1. 為什麼要改？（ZooKeeper 的三大痛點）

在舊架構中，Kafka 像是一個需要「外掛大腦」的機器人。ZooKeeper 負責幫 Kafka 存儲元數據（Metadata，例如：誰是 Leader、有哪些 Topic 等）。但這帶來了幾個嚴重的問題：

#### A. 運維的雙重負擔
你必須同時維護兩套完全不同的系統。
* **問題**：兩者的配置邏輯、監控指標、安全性設定（如證書）完全不同。在 K8s 上，你得跑兩組 StatefulSet。

#### B. 分區數量的「天花板」 (Scalability)
ZooKeeper 有一個硬傷：它能處理的元數據量有限。
* **現象**：當一個 Kafka 叢集的分區 (Partition) 數量達到幾十萬個時，ZooKeeper 會變得非常吃力，導致整個叢集變慢。這限制了大型企業擴張叢集的能力。

#### C. 災難復原太慢 (Controller Failover)
如果 Kafka 的「指揮官 (Controller)」掛掉了，Kafka 必須去 ZooKeeper 那邊重新選舉一個新的指揮官，並從 ZooKeeper 把所有的資料重新載入一次。
* **痛點**：在資料量很大的時候，這個過程可能長達幾分鐘，這幾分鐘內 Kafka 是無法工作的。

---

### 2. 什麼是 KRaft？（Kafka 的自癒進化）

**KRaft** 的核心思想是：**「Kafka 不需要外掛大腦，它自己就是大腦。」**



#### 如何運作？
在 KRaft 模式下，Kafka 會選出幾台 Broker 擔任 **Controller** 角色（這就是你剛才 YAML 檔案中 `controller` 池的作用）。
* 這些 Controller 組成一個小型的 **Quorum**。
* 它們使用 **Raft 一致性演算法** 來同步元數據。
* **關鍵差異**：元數據現在被存儲在 Kafka 內部的一個特殊 Topic 中。

---

### 3. KRaft 帶來的優勢（研究結論）

| 特性 | ZooKeeper 架構 (舊) | KRaft 架構 (新) |
| :--- | :--- | :--- |
| **系統組件** | Kafka + ZooKeeper (兩套系統) | 只有 Kafka (一套系統) |
| **分區上限** | 有限 (通常受限於 ZK 效能) | 極高 (可達百萬級甚至更多) |
| **故障恢復** | 較慢 (需要從 ZK 重新讀取) | **極快** (元數據已在內存中同步) |
| **K8s 部署** | 複雜 (需配置兩個 StatefulSet) | **簡單** (只需配置 NodePools) |

---

### 4. 對你 EKS 研究的影響

如果你在 EKS 上使用 KRaft（如你提供的那份 YAML）：
1.  **省錢**：你不需要為了跑 ZooKeeper 而多開 3 個 EKS 節點。
2.  **更穩**：減少了「Kafka 跟 ZooKeeper 連線斷掉」這種網路靈異事件發生的機率。
3.  **現代化**：主流社群與 Strimzi 都在全力支援 KRaft，未來 ZooKeeper 版本的 Kafka 將會停止維護。

> **一句話總結：**
> ZooKeeper 像是 Kafka 必須背著跑的「外部行動電源」，而 KRaft 是讓 Kafka 內建了「大容量電池」。

**你剛才那份 YAML 裡分了 `controller` 和 `broker` 兩個池，這正是 KRaft 最標準的做法。你有打算實際在 EKS 上執行 `kubectl apply` 看看這 6 個 Pod (3控3代) 跑起來的樣子嗎？**
